## 函数声明式UI
`Compose`的声明式UI使用函数来实现，以显示文字为例：
```kotlin
class MainActivity : ComponentActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()

        setContent {
            // 显示文字：Hello Compose
            Text("Hello Compose")
        }
    }
}
```
这样就会在Activity中显示“Hello Compose”。

我们来尝试在3秒后修改文字刷新UI，代码如下：
```kotlin
// ❌ 无法实现需求的例子
class MainActivity : ComponentActivity() {
    // 定义一个普通字符串变量
    var text = "Hello Compose"

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()

        setContent {
            Text(text)
        }

        // 延迟3秒，修改文字
        lifecycleScope.launch {
            delay(3000)
            text = "Changed Text"
        }
    }
}
```
我们会发现文字并不能刷新。我们可以联想到Flutter中修改数据后需要调用`setState`才能刷新UI，也就是说需要一种方式让声明式UI框架触发刷新。`Compose`并不会自动追踪上面的普通`text`字符串变量的变化，要实现刷新UI，需要借助`Compose`中提供的`State`类型。

### State
`State`类型定义如下，其实就是可以从中读取值`value`：
```kotlin
@Stable
interface State<out T> {
    val value: T
}
```

如果我们想要修改值，则需要使用`State`的子类`MutableState`：
```kotlin
@Stable
interface MutableState<T> : State<T> {
    override var value: T
    operator fun component1(): T
    operator fun component2(): (T) -> Unit
}
```

`Compose`会我们提供了`mutableStateOf`方法（还有很多类似的一系列方法）来便捷地创建`MutableState`。现在我们利用它来实现刷新UI：
```kotlin
class MainActivity : ComponentActivity() {

    // 通过 mutableStateOf 来构造 MutableState 变量
    val text = mutableStateOf("Hello Compose")

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()

        setContent {
            // 通过 MutableState 的 value 读取值
            Text(text.value)
        }

        lifecycleScope.launch {
            delay(3000)
            // 通过 MutableState 的 value 写入值
            text.value = "Changed Text"
        }
    }
}
```
如此就可以达到更新UI的效果了。

声明 MutableState 对象的方法还可以使用`by`委托语法或者解构声明：
```kotlin
var mutableState by mutableStateOf("") // 使用 by 的情况，就可以直接使用变量，不需要再使用 value 属性

val (value, setValue) = remember { mutableStateOf(default) }
```

`by` 委托语法需要以下导入：
```kotlin
import androidx.compose.runtime.getValue
import androidx.compose.runtime.setValue
```

> `Compose`通过编译器插件和运行时实现，它内部会自动监听`State`类型的数据变化，具体的原理在源码分析中再了解

> 比如`Flow`、`LiveData`都可以转换为`State<T>`

### remember
#### 不使用 remember 产生的问题
多数情况，我们会把状态变量放到实际需要使用它的`@Composable`函数或者Lambda中，而不是全局：
```kotlin
// ❌ 错误例子：
class MainActivity : ComponentActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()

        setContent {
            TextItem()
        }
    }

    // 创建一个 @Composable 函数
    @Composable
    fun TextItem() {
        // 将创建 MutableState 的代码放在 @Composable 函数中
        var text by mutableStateOf("Hello Compose")
        Text(text)
    
        // 这里只是简单例子，所以在这里启动协程，但实际不建议这样写
        lifecycleScope.launch {
            delay(3000)
            text = "Changed Text"
        }
    }

}
```
**此时我们会发现刷新UI失效了。**

我们分析一下原因：声明式 UI 的写法，刷新UI的方式就是重新执行 UI 的声明，对于`Compose`来说就是会再次执行`@Composable`函数，这个过程就叫重组，所以`Compose`相比Android传统的 View 命令式 UI 增加了一个`重组`流程。既然`TextItem()`被再次执行，那么`text`变量也会重新创建，而更新的数据是原本的变量，也就导致无法刷新UI。


我们可以尝试通过一个简单但**不建议**的方式简单解决这个问题：因为`Compose`的重组会尽量只重新执行实际需要刷新的范围，在上面的错误例子中，`text`变量直接在`TextItem()`重组作用域中使用，所以会**重组时会再次调用`TextItem()`函数**，原本的`text`变量就会失效并创建新的；而如果我们将`text`变量创建在它使用的重组作用域之外，`Compose`在重组时就只会执行使用`text`的函数，就可以避免重新创建`text`变量。下面看看代码：
```kotlin
@Composable
fun TextItem() { // --- TextItem 重组作用域
    var text by mutableStateOf("Hello Compose")
    // 包裹一个层级 Surface
    Surface { // --- Surface 重组作用域
        Text(text)
    }

    lifecycleScope.launch {
        delay(3000)
        text = "Changed Text"
    }
}
```
此时由于`text`只在`Surface`的重组作用域中使用，所以`Compose`在重组时只会执行`Surface`的lambda函数，`text`变量就不会重新创建。（**但这并不是合理的解决方式，下面我们将使用`remember`来解决**）

> 例如 `Row` 和 `Column` 都不能创建重组作用域，因为它们都是`inline`函数，所以这里选择使用`Surface`来作为例子

#### 使用 remember 解决
`Compose`提供了`remember`函数，通过它可以让`Compose`帮我们缓存变量，即使重组也不会重新创建。
```kotlin
@Composable
fun TextItem() {
    val name = remember {  mutableStateOf("Hello Compose") }
    Text(name.value)

    lifecycleScope.launch {
        delay(3000)
        name.value = "Changed Text"
    }
}
```
只要在`@Composable`函数中创建并使用的可变状态，都应该使用`remember`函数来避免受到重组影响。

#### 使用 key 的 remember 函数
`remember`可以在`@Composable`函数中缓存数据，避免重新创建。但有的情况下，我们会根据一些条件来决定是缓存还是创建新的，此时就需要使用带`key`参数的`remember`函数：
```kotlin
@Composable
fun UserProfile(userId: String) {
    // 当 userId 变化时，重新获取数据
    val userData by remember(userId) {
        mutableStateOf(fetchUserData(userId))  // 模拟网络请求或计算
    }

    Text("用户：${userData?.name ?: "加载中..."}")
}
```
如果`userId`变化，应该重新获取数据，所以我们使用`userId`作为`remember`函数的key。

再看一个例子，在循环中用 `remember(key)` 让每个 item 有独立的展开状态，避免状态错乱：
```kotlin
@Composable
fun SimpleExpandableList() {
    val items = listOf("苹果", "香蕉", "橙子", "葡萄")

    Column {
        items.forEach { fruit ->
            var expanded by remember(fruit) { mutableStateOf(false) }

            Row(
                modifier = Modifier
                    .fillMaxWidth()
                    .clickable { expanded = !expanded }
                    .padding(12.dp)
            ) {
                Text(fruit)
                Spacer(Modifier.weight(1f))
                Text(if (expanded) "收起" else "展开")
            }

            if (expanded) {
                Text(
                    "这是 $fruit 的详细信息",
                    modifier = Modifier.padding(start = 32.dp, bottom = 8.dp)
                )
            }
        }
    }
}
```

#### 其他 remember
1. **rememberUpdatedState**

在协程或回调中，如果直接捕获 `@Composable` 中的状态变量，会形成 “闭包快照”，即使状态更新，闭包内仍使用旧值。此时就可以使用`rememberUpdatedState`来解决这个问题：
```kotlin
    @Composable
    fun MyTimer() {
        var count by remember { mutableStateOf(0) }

        // ✅ 创建一个 always-updated reference
        val currentCount by rememberUpdatedState(count)

        LaunchedEffect(Unit) {
            while (true) {
                delay(1000)
                println("Count is:  $currentCount") // 打印最新值
            }
        }

        Button(onClick = { count++ }) { Text("Count:  $count") }
    }
```

可以看到 `rememberUpdatedState` 源码，重组时重新执行它，都会设置一次最新值
```kotlin 
@Composable
fun <T> rememberUpdatedState(newValue: T): State<T> = remember {
    mutableStateOf(newValue)
}.apply { value = newValue }
```


2. **rememberCoroutineScope**
`rememberCoroutineScope`：提供协程作用域，生命周期跟随当前重组作用域


## Modifier
在使用很多组件时，都可以传入`Modifier`参数，比如用于设置内边距`padding`、背景`background`、点击事件`clickable`、宽高等。`Compose`把通用的属性设置都放到`Modifier`中。

`Modifier` 是一个组合式的接口，各个 `Modifier` 的扩展方法，实际都是增加一个 `Modifier.Element`，类似协程的Context设计。内部怎么使用的？比如 `Modifier.padding` 会构造 `PaddingNode`，实现了`MeasureScope.measure`方法，会处理布局测量。又比如 `Modifier.background` 会构造 `BackgroundNode` ，它实现了`ContentDrawScope.draw()`，会介入绘制阶段。

从本质上来说，`Modifier`就是提供各种`LayoutModifierNode`（布局）、`DrawModifierNode`（绘制）、`PointerInputModifierNode`（触摸事件）等功能实现的容器，`Compose`会从`Modifier`中取出它们用于组件的各个流程。（对比传统View的开发，把依靠继承重写的方式改为了组合的方式）

## CompositionLocal
`CompositionLocal` 是 `Compose` 提供的一种作用域限定的隐式数据传递机制。通过它可以在组合树（Composition Tree）的某个节点提供一个值，其所有后代组件都可以通过 `.current` 直接读取，无需参数传递。

下面看一下`CompositionLocal`的使用例子，需要配合`CompositionLocalProvider`使用：
```kotlin
// 推荐命名：以 Local 开头，便于识别
// 通过。compositionLocalOf 创建子类型 ProvidableCompositionLocal
val LocalUser = compositionLocalOf<User?> { null }
// staticCompositionLocalOf 对应不变的数据

@Composable
fun App(user: User) {
    // 使用 provides 中缀表达式关联 LocalUser 和 user
    CompositionLocalProvider(LocalUser provides user) {
        // 所有子组件现在都能访问 LocalUser.current
        HomeScreen()
    }
}

// 在任意后代中读取
@Composable
fun UserProfile() {
    val user = LocalUser.current
    if (user != null) {
        Text("Welcome,  $ {user.name}")
    }
}
```
在`CompositionLocalProvider`中，使用了一个奇怪的写法，这个`provides`是 `compositionLocalOf` 方法返回的`ProvidableCompositionLocal`类型提供的中缀运算符，用于关联`LocalUser`和实际的数据对象`user`。之后就可以通过`LocalUser.current`来获取到`user`对象。

### CompositionLocal 的适用场景
多个层级的`CompositionLocal`，更近的层级的数据优先级更高，所以更适合通用的上下文场景。比如主题颜色在各个组件都可能会用到，如果每次都手动传参，代码会非常冗余，这种情况就很适合使用`CompositionLocal`机制。实际上`Compose`提供的`MaterialTheme`内部就使用了`CompositionLocal`。

> `CompositionLocal`是自上而下的，而 Compose 中没有类似 Flutter Notification 这样自下而上的机制。这种情况一般使用回调函数，把事件转化为状态变更，或者借助`ViewModel`等方式。

> 在 @Composable 函数中，我们可以使用 `LocalActivity`、`LocalContext` 等方式便捷获取当前上下文，因为 `setContent` 内部会构造一个`CompositionLocalProvider`把我们的 content 包裹住。可以在源码分析中具体了解。


## Side Effect 副作用
需要在 `Composable` 生命周期内执行一次/多次的操作都属于`Side Effect`，`Compose`为我们提供了三种`Side Effect`，下面依次介绍它们。

### LaunchedEffect
在首次进入 `Composition` 或者 `key` 变化时重启，并且会自动取消协程。
```kotlin
@Composable
fun UserProfile(userId: String) {
    var user by remember { mutableStateOf<User?>(null) }

    // 当 userId 变化时，重新启动协程获取数据
    LaunchedEffect(userId) {          // ← key = userId
        user = withContext(Dispatchers.IO) {
            api.fetchUser(userId)     // 网络请求
        }
    }

    // 或者监听 Flow（最经典用法）
    LaunchedEffect(Unit) {            // Unit 表示只执行一次
        viewModel.userFlow.collect { user = it }
    }

    Text(user?.name ?: "加载中...")
}
```

### DisposableEffect
专门用于需要清理（cleanup）的副作用，比如注册监听器、开启定时器、订阅广播等。
```kotlin
@Composable
fun LocationTracker() {
    val locationManager = remember { context.getSystemService(LocationManager::class.java) }

    DisposableEffect(Unit) {   // key 可以是 Unit（只执行一次）或某个变化的参数
        val listener = LocationListener { location ->
            // 更新位置
        }

        locationManager.requestLocationUpdates(
            LocationManager.GPS_PROVIDER, 0L, 0f, listener
        )

        // 返回清理函数（必须）
        onDispose {
            locationManager.removeUpdates(listener)
            println("已注销位置监听")
        }
    }

    Text("正在跟踪位置...")
}
```

### SideEffect
每次重组后都执行一次

```kotlin
@Composable
fun AnalyticsScreen(
    screenName: String,
    eventLogger: AnalyticsLogger
) {
    SideEffect {
        // 每次重组成功（UI 真正更新后）都记录
        eventLogger.logScreenView(screenName)
    }

    // 或者同步状态到外部对象
    val count by viewModel.count.collectAsState()
    SideEffect {
        externalSystem.updateCounter(count)   // 非 Compose 代码
    }

    Text("计数: $count")
}
```

## 监听 Android 生命周期
`Compose`中虽然可以使用`LaunchedEffect`和`DisposableEffect`满足大多数场景生命周期的处理，但有些情况仍然需要使用 Android 的 `onCreate`、`onStart`、`onResume`等生命周期。可以使用以下一些方式：


### Flow.collectAsStateWithLifecycle
```kotlin
@Composable
fun MyScreen(viewModel: MyViewModel) {

    // 方式一：收集 Flow，只在 STARTED 时消费（最常见、最安全）
    val data by viewModel.dataFlow.collectAsStateWithLifecycle()

    Text("当前数据: $data")
}
```

### lifecycleOwner 监听
```kotlin
@Composable
fun LifecycleAwareComposable() {
    val lifecycleOwner = LocalLifecycleOwner.current

    DisposableEffect(lifecycleOwner) {
        val observer = LifecycleEventObserver { _, event ->
            when (event) {
                Lifecycle.Event.ON_CREATE   -> println("ON_CREATE")
                Lifecycle.Event.ON_START    -> println("ON_START → 开始监听位置/传感器/动画")
                Lifecycle.Event.ON_RESUME   -> println("ON_RESUME → 恢复交互、刷新数据")
                Lifecycle.Event.ON_PAUSE    -> println("ON_PAUSE → 暂停动画、保存临时状态")
                Lifecycle.Event.ON_STOP     -> println("ON_STOP → 停止定位、注销广播")
                Lifecycle.Event.ON_DESTROY  -> println("ON_DESTROY → 清理资源")
                else -> {}
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        // 清理（必须）
        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
            println("已移除 LifecycleObserver")
        }
    }

    Text("监听生命周期中...")
}
```

### repeatOnLifecycle
在指定生命周期执行协程：
```
@Composable
fun MyScreen(viewModel: MyViewModel) {
    val lifecycleOwner = LocalLifecycleOwner.current

    // 手动 repeatOnLifecycle（更灵活，可执行任意协程代码）
    LaunchedEffect(lifecycleOwner) {
        lifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            // 这里相当于 onStart / onResume
            launch {
                // 比如监听事件、开始轮询、收集其他 Flow
                viewModel.someEventFlow.collect { event ->
                    // 处理事件
                }
            }

            // 模拟一次性操作（只在进入 STARTED 时执行一次）
            launch {
                // 相当于 onResume 做的事
                println("屏幕可见，开始加载或刷新")
            }
        }
        // 当离开 STARTED（onPause / onStop）时，自动取消上面的协程
    }

    Text("当前数据: $data")
}
```

## key
虽然 Compose 会自动为每个 @Composable 函数内生成不同的 key，但假如在列表组件中多次调用同一个 @Composable 函数，它们的 key 就是一样的，只能通过位置来区分。假如数据发生增删和移动操作，Compose runtime 无法区分，就可能错误地复用状态。当然，状态由外部数据作为参数传入的不会有问题，而是 `remember`、`Effect` 这些附着于当前组件生命周期的状态可能被错误地复用。

所以在列表之类的场景，建议手动设置局部唯一的 key。

例如这个场景，列表初始数据为 `1,2,4`，两秒之后插入 `3`，但我们没有设置 key：
```kotlin
@Composable
fun MyList() {
    var list by remember { mutableStateOf(listOf("1", "2", "4")) }
    LazyColumn {
        items(list) { item ->
            MyText(item)
        }
    }

    LaunchedEffect(Unit) {
        delay(2000)
        list = listOf("1", "2", "3", "4")
    }

}

@Composable
fun MyText(text: String) {
    SideEffect {
        println("MyText Side：$text")
    }
    LaunchedEffect(Unit) {
        println("MyText Launched：$text")
    }

    Text(text)
}
```

打印结果为：
```
MyText Side：0 : 0
MyText Side：1 : 1
MyText Side：2 : 3
MyText Launched：0 : 0
MyText Launched：1 : 1
MyText Launched：2 : 3

// 两秒后
MyText Side：2 : 2
MyText Side：3 : 3
MyText Launched：3 : 3
```
因为没有设置 key，所以`0,1,3`变为`0,1,2,3`，会根据位置匹配，所以会当作 index = 2 的数据 3 更新为数据 2，并且新增 index = 3 的数据 3。

而我们以数据本身作为 key ，而不是位置作为 key 的情况下：
```
// 省略其他代码
itemsIndexed(list, key = { _, item -> item }) { index, item ->
    MyText("$index : $item")
}
```

```
MyText 更新：0 : 0
MyText 更新：1 : 1
MyText 更新：2 : 3
MyText 创建：0 : 0
MyText 创建：1 : 1
MyText 创建：2 : 3

// 两秒后
MyText 更新：3 : 3
MyText 更新：2 : 2
MyText 创建：2 : 2
```
现在可以正确地认为，数据为 3 的 item 只是更新（布局位置移动），而数据为 2 的 item 被创建，并且放到了 index = 2 的位置。

## 路由导航
`Navigation2`的基本用法：
```kotlin
// Compose 导航支持多种类型，比如我们这里自定义类型（或者直接使用字符串等都可以）
@Serializable
sealed interface Route {

    @Serializable
    data object Home : Route

    @Serializable
    data object ProfileList : Route

    @Serializable
    data class ProfileDetail(
        val userId: Long,
        val name: String? = null   // 可选参数
    ) : Route

    @Serializable
    data class Settings(
        val openSection: String = "general"  // 默认值
    ) : Route
}


@Composable
fun AppNavigation() {
    val navController = rememberNavController()

    NavHost(
        navController = navController,
        startDestination = Route.Home
    ) {
        composable<Route.Home> {
            HomeScreen(
                onGoToProfileList = { navController.navigate(Route.ProfileList) },
                onGoToSettings = { navController.navigate(Route.Settings()) }
            )
        }

        composable<Route.ProfileList> {
            ProfileListScreen(
                onProfileClick = { userId, name ->
                    navController.navigate(Route.ProfileDetail(userId, name))
                }
            )
        }

        composable<Route.ProfileDetail> { backStackEntry ->
            val detail: Route.ProfileDetail = backStackEntry.toRoute()
            ProfileDetailScreen(
                userId = detail.userId,
                name = detail.name,
                onBack = { navController.popBackStack() }
            )
        }

        composable<Route.Settings> { backStackEntry ->
            val settings: Route.Settings = backStackEntry.toRoute()
            SettingsScreen(
                initialSection = settings.openSection,
                onSave = { navController.popBackStack() }
            )
        }
    }
}

// 页面例子：
@Composable
fun HomeScreen(
    onGoToProfileList: () -> Unit,
    onGoToSettings: () -> Unit
) {
    Column {
        Button(onClick = onGoToProfileList) { Text("查看用户列表") }
        Button(onClick = onGoToSettings) { Text("设置") }
    }
}
```
`composable`是`NavGraphBuilder`的扩展函数，用于添加路由


## 实战须知
### ViewModel、CompositionLocal
如果使用 ViewModel 中的 `flow.collectAsState` 转换为 State，就不需要使用 `remember`，因为依靠 `ViewModel` 可以得到同一个 flow 对象。而 Compose 函数中的临时状态变量需要使用 `remember` 函数，因为需要自己主动维护它的引用

Compose中一般使用 ViewModel（主要的）、CompositionLocal（适合不涉及业务逻辑的轻量共享） 来共享状态。

### 一些简单优化
#### 使用特定类型的 State
例如状态是 Int 类型，就可以使用 `mutableIntState`，不用装箱拆箱，性能更好。其他的一些基本类型有对应的特定方法

#### lambda 延迟执行
Compose 可以仅触发布局/绘制，而不触发重组，从而提升性能。通过将状态读取延迟到布局阶段，可以在这个过程中避免进行重组。
```kotlin
val listState = rememberLazyListState()

Image(
    // ...
    // Non-optimal implementation!
    Modifier.offset(
        with(LocalDensity.current) {
            // State read of firstVisibleItemScrollOffset in composition
            (listState.firstVisibleItemScrollOffset / 2).toDp()
        }
    )
)

// 改为lambda，也就是让值在布局阶段才读取，从而只重新布局，而跳过不必要的重组
val listState = rememberLazyListState()

Image(
    // ...
    Modifier.offset {
        // State read of firstVisibleItemScrollOffset in Layout
        IntOffset(x = 0, y = listState.firstVisibleItemScrollOffset / 2)
    }
)
```

#### derivedStateOf 
`derivedStateOf`可以用于满足一定条件才会触发重组，提升性能。类似 Flutter Provider 的 Selector

#### 跳过中间层
跨多层传递，中间层不需要监听，可以使用 lambda 让子控件读取，避免中间层执行不必要的重组
```kotlin
@Composable
@Preview
fun Example3() {
    //...
    Column {
        //                                           👇🏻使用lambda让子控件读取
        ScrollStateChecker(scrollValueProvider = { scrollState.value })
        Column(
            //...
        ) {
            //...
        }
    }
}

@Composable
private fun ScrollStateChecker(scrollValueProvider: () -> Int) {
    //                          👇🏻使用lambda读取
    Text("scrollValue:${(scrollValueProvider())}")
}
```

### 一些注解
* `@NonRestartableComposable` 标记该 Composable 不可单独重启，重组作用域合并到父组件中，目的是减少重组调度开销，优化性能。
* `@NonSkippableComposable`：标记不可跳过重组
